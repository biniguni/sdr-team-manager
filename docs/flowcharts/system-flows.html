SDR Team Manager 시스템 흐름도

공개 조회, 승인 편집자 쓰기, 경기 결과 전용 권한을 기준으로 현재 앱의 시스템 아키텍처와 주요 사용자 플로우를 정리한 문서입니다.

1. 시스템 아키텍처

사용자는 Vercel에 배포된 Next.js 앱을 통해 데이터를 조회하고, 쓰기 작업은 Server Action과 Supabase RLS를 모두 통과해야 합니다.

공개 조회 승인 편집자 쓰기 결과 권한 분리
사용자 브라우저 대시보드, 라인업, 선수, 시즌, 랭킹 화면 조회
편집자 로그인 /login에서 Supabase Auth 이메일/비밀번호 로그인
→
Vercel Next.js App Router 앱 호스팅
src/proxy.ts 현재는 공개 조회 모드라 전역 로그인 차단 없음
→
Server Components 페이지 진입 시 Supabase에서 공개 데이터를 읽어 UI 구성
Client Components 필터, 폼, 드래그앤드롭 라인업, 모바일 메뉴 처리
→
Server Actions 쓰기 요청의 권한 확인, 검증, 저장, 페이지 갱신 담당
권한 헬퍼 requireEditor, requireMatchResultManager
→
Supabase PostgreSQL players, seasons, matches, match_roster, period_lineups 등 저장
RLS + team_editors SELECT는 공개, 쓰기는 승인 편집자만 허용

제품 영향: 방문자는 로그인 없이 팀 데이터를 볼 수 있고, 실제 데이터 변경은 앱 권한 검사와 DB RLS를 모두 통과해야 하므로 운영 실수를 줄입니다.

2. 권한 및 접근 흐름

로그인 여부만으로 쓰기 권한이 생기지 않습니다. Supabase Auth로 사용자를 확인한 뒤, team_editors allowlist와 결과 관리 권한을 따로 확인합니다.

1 사용자 진입 로그아웃 방문자도 공개 페이지 조회 가능
2 쓰기 시도 UI 버튼은 숨겨지지만 보안 기준은 Server Action과 RLS
3 승인 편집자 확인 team_editors에 있는 사용자만 일반 데이터 수정 가능
4 결과 권한 확인 스코어, MOM, 선수 기록은 can_manage_match_results 필요
5 DB 정책 통과 Supabase RLS가 최종 쓰기 권한을 다시 제한
사용자 유형 허용 작업
공개 방문자 대시보드, 선수, 경기, 라인업, 랭킹 조회
미승인 로그인 사용자 공개 조회만 가능, 쓰기 불가
일반 편집자 선수, 시즌, 스쿼드, 경기 일정, 포메이션, 라인업 관리
결과 관리자 일반 편집 + 스코어, 완료 상태, MOM, 선수 기록 저장
보안 기준 현재 결정
전역 로그인 차단 꺼짐. 공개 조회 제품 방향 유지
개인정보/메모 공개 모드에서 앱 사용 및 노출 금지
서비스 롤 키 브라우저와 저장소에 노출 금지
RLS SELECT 공개, INSERT/UPDATE/DELETE 승인 편집자 제한

3. 사용자 플로우

현재 제품의 주요 업무는 시즌 준비, 경기 생성, 라인업 편성, 경기 결과 기록, 대시보드 확인 순서로 이어집니다.

공개 방문자
대시보드 조회 시즌 기록과 최근 경기 확인
선수/랭킹 확인 공개 안전 필드만 표시
일반 편집자
선수 등록 이름, 등번호, 발 점수, 활성 상태 관리
시즌/스쿼드 구성 시즌별 참가 선수 관리
경기 일정 생성 상대, 날짜, 홈/원정, 기간 구성
라인업 준비 경기 로스터와 기간별 포지션 배정
결과 관리자
경기 결과 입력 스코어, 완료 상태, MOM 저장
선수 기록 입력 출전, 득점, 도움, 경고, 퇴장 저장

4. 라인업 준비 파이프라인

현재 UI 검토의 중심입니다. 정식 선수와 경기 전용 게스트를 모두 match_roster에 모은 뒤, period_lineups에 배정합니다.

1 /lineup 진입 활성 시즌 경기만 보여주고 기본 경기 자동 선택
2 경기 로스터 구성 시즌 스쿼드에서 등록 선수 추가, 경기 전용 게스트 추가
3 기간/포메이션 선택 1Q~4Q 또는 전반/후반, 4-2-3-1 등 선택
4 포지션 배정 피치 보드 또는 우측 포지션 패널에서 배정
5 저장 및 집계 갱신 중복 검증 후 period_lineups 저장, position_performance 갱신
등록 선수 players → squad_members → match_roster → period_lineups
경기 전용 게스트 match_roster에 guest_name, guest_number만 저장
현재 제한 게스트는 라인업 배정용. 기록, MOM, 랭킹은 등록 선수 기준

5. 쓰기 작업 공통 순서

선수, 시즌, 경기, 포메이션, 라인업, 기록 저장은 모두 이 순서를 기본으로 따릅니다.

1 권한 확인 일반 편집자 또는 결과 관리자 권한 확인
2 입력 검증 필수값, 중복, 음수 기록, 기간/포지션 관계 확인
3 Supabase 저장 RLS 정책을 통과한 요청만 DB 반영
4 관련 데이터 갱신 라인업 저장 후 포지션 집계 등 파생 데이터 갱신
5 페이지 재검증 revalidatePath로 화면에 최신 데이터 반영

제품 영향: UI에서 실수로 잘못된 값을 보내도 서버와 DB가 다시 확인하므로, 공개 조회 환경에서도 운영 데이터가 더 안정적으로 유지됩니다.

6. 데이터 관계 흐름

라인업과 결과 기록이 어느 테이블을 거쳐 대시보드와 랭킹으로 이어지는지 정리했습니다.

players 등록 선수, 등번호, 발 점수, 활성 상태
→
squad_members 선수가 특정 시즌 스쿼드에 속함
→
seasons 시즌 단위로 경기, 스쿼드, 집계 분리
→
matches + periods 경기와 경기 안의 1Q~4Q 또는 전후반
→
match_roster 해당 경기에서 쓸 수 있는 선수와 게스트
formations 4-4-2, 4-3-3, 3-5-2, 4-2-3-1
→
position_slots 포메이션별 포지션 코드와 좌표
→
period_lineups 기간별 포지션과 match_roster 참가자 연결
→
position_performance 시즌별 선수 포지션 출전 집계
→
dashboard 요약, 개인 기록, 경기 히스토리 표시
matches 스코어, 완료 상태, MOM
→
player_match_stats 출전, 득점, 도움, 카드 기록
→
ranking 선수별 출전, 득점, 도움, 전체 경기 MOM 순위와 개인 상세

7. 배포 및 DB 적용 순서

기존 Supabase DB와 Vercel 배포를 맞출 때의 운영 순서입니다. 새 DB는 schema.sql을 기준으로 시작하고, 기존 DB는 필요한 migration SQL 적용 여부를 먼저 확인합니다.

1 로컬 검증 npm.cmd run lint, npm.cmd run build
2 DB 스키마 확인 match_roster, match_roster guest 필드, period_lineups.match_roster_id
3 RLS 확인 public read, approved-editor write 정책 적용
4 Vercel 환경변수 NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY 설정
5 권한 스모크 테스트 로그아웃, 미승인 사용자, 일반 편집자, 결과 관리자 확인
기준 문서: docs/specs/design.md, docs/specs/requirements.md, docs/security.md, docs/specs/progress.md. 생성 위치: docs/flowcharts/system-flows.html
