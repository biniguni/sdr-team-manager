## 요약

Samala는 경기 기록 앱이라기보다 클럽 운영 전체를 다루는 모바일 중심
운영 플랫폼이다. 경기, 선수, 전술, 개인 기록뿐 아니라 클럽 생성/가입,
초대 링크, 가입 신청 승인, 운영진 역할, 회비, 참석 투표, 도착 체크,
댓글, 공지, 미션, 좋아요 투표, 공유 이미지 생성까지 포함한다.

SDR Team Manager는 현재 단일 팀의 시즌, 선수, 경기, 라인업, 결과,
통계, 랭킹을 안정적으로 관리하는 쪽에 더 가깝다. 우리 앱은
Next.js/Supabase 기반이고, 공개 조회와 승인 편집자 쓰기 권한을 명확히
분리한다. Samala는 Firebase 클라이언트 앱 구조라 기능 확장은 빠르지만,
데이터 권한과 개인정보 노출은 Firebase Rules에 크게 의존한다.

추천 기본 방향:

1. Samala의 운영 편의 기능 중 `참석 투표`, `도착 체크`, `희망 포지션`,
   `상대팀 관리`, `공유 이미지 생성`은 우리 앱에 맞게 도입할 가치가 높다.
2. `회비`, `연락처`, `출생년도`, `사진`, `댓글`, `공지`, `회칙`은 공개 조회
   정책과 충돌할 수 있으므로 별도 private 영역과 RLS 정책을 먼저 설계해야 한다.
3. `공개 회원가입`, `익명 체험`, `멀티 클럽 검색/가입`은 현재 제품 목표와
   다르므로 당장 따라 하지 않는 편이 낫다.

| 유지보수성 | 한 페이지/대형 JS에 기능이 집중 | 기능별 route/action/component 분리 |

제품 영향:

- Samala는 실제 팀 운영자의 반복 작업을 많이 덜어준다. 특히 경기 전
  참석 확인, 정원 관리, 도착 체크, 희망 포지션 수집은 라인업 품질에
  직접 연결된다.
- 우리 앱은 데이터 모델과 보안 경계가 더 명확하다. 따라서 Samala식
  기능을 가져오더라도 데이터는 Supabase 테이블로 분리하고 권한은
  RLS와 Server Action에서 막는 방식이 맞다.

## 참고하면 좋은 점

### 1. 경기 전 운영 흐름

Samala의 가장 강한 부분은 경기 전 운영이다.

- 경기별 참석/불참/미정 투표
- 참석 정원 제한
- 참석 마감 시간
- 용병 수 표시
- 도착 체크
- 지각 기준 설정
- 희망 포지션 선택
- 경기별 댓글

우리 앱에 적용하면 라인업 화면의 입력 품질이 좋아진다. 현재 `/lineup`은
편집자가 match roster를 먼저 구성해야 하는데, 참석 투표가 있으면
로스터 후보를 자동 추천할 수 있다.

추천 기본값:

- 먼저 `참석/불참/미정`과 `희망 포지션`만 추가한다.
- `도착 체크`, `정원`, `마감 시간`은 다음 단계로 둔다.
- 댓글은 개인정보/분쟁/삭제 정책이 필요하므로 후순위로 둔다.

### 2. 상대팀 데이터베이스

Samala는 상대팀 탭에서 상대팀 정보와 전적을 따로 관리한다.

우리 앱은 현재 `matches.opponent`가 문자열 중심이다. 상대팀을 별도
테이블로 만들면 같은 상대를 여러 방식으로 입력하는 문제가 줄고,
상대팀별 전적/득실/승률을 만들기 쉬워진다.

추천 기본값:

- `opponents` 테이블을 추가한다.
- 안전한 공개 필드만 둔다: `name`, `area`, `uniform_color`, `notes_public`.
- 내부 메모가 필요하면 `opponent_private_notes`처럼 editor-only 테이블로
  분리한다.

### 3. 공유 이미지 생성

Samala는 `html2canvas`로 경기 카드, 시즌 하이라이트, 팀 소개 이미지를
생성하고 Web Share API 또는 다운로드로 공유한다.

우리 앱에 적용하면 경기 후 공유와 라인업 공유가 쉬워진다. 특히 모바일에서
캡처하기 어려운 긴 화면보다, 앱이 정해진 비율의 이미지를 만들어 주는 편이
사용자 경험이 좋다.

추천 기본값:

- 1차: 라인업 이미지 저장/공유
- 2차: 경기 결과 카드
- 3차: 시즌 요약 카드

### 4. 모바일 앱 같은 사용감

Samala는 PWA manifest와 설치 안내가 있고, 탭/팝업 중심으로 모바일 앱처럼
동작한다. 서비스 워커는 요청을 그대로 `fetch`하는 형태라 실질적인 오프라인
캐시는 없지만, 설치형 UX는 갖추고 있다.

우리 앱도 모바일 사용자가 많다면 manifest와 아이콘, 설치 안내는 고려할
수 있다. 다만 오프라인 지원은 별도 요구가 생기기 전까지는 하지 않는 것이
좋다.

## 스펙 및 배포 관점

### Samala에서 확인한 스펙

| 항목 | 확인 내용 |
| --- | --- |
| 호스팅 | 응답 헤더에 `x-fh-requested-host` 계열 흔적이 있어 Firebase Hosting일 가능성이 높다. |
| 데이터/Auth | Firebase Auth, Realtime Database, Storage 사용 |
| 프론트엔드 | 정적 HTML + CSS + `main.js` |
| 외부 라이브러리 | Chart.js, gif.js, html2canvas, lucide |
| PWA | `manifest.json` 있음, `display: standalone`, 앱 아이콘 지정 |
| 서비스 워커 | 설치 이벤트와 fetch pass-through만 있음. 오프라인 캐싱은 사실상 없음 |
| 보안 헤더 | HSTS는 확인됨. CSP, X-Frame-Options, Referrer-Policy 등은 응답 헤더에서 확인되지 않음 |
| 캐시 | HTML `Cache-Control: max-age=3600` |

### 우리 앱에 반영할 스펙 제안

| 영역 | 권장 기본값 |
| --- | --- |
| PWA | manifest와 아이콘부터 추가. 오프라인 캐시는 보류 |
| 공유 이미지 | 서버 저장 없이 브라우저에서 PNG 생성 후 공유/다운로드 |
| 참석 기능 | Supabase 테이블로 명확히 분리 |
| 배포 | 기존 Vercel + Supabase 유지 |
| 보안 헤더 | Vercel `next.config.ts`에서 CSP/Referrer-Policy/Permissions-Policy 검토 |
| 라이브러리 | 공유 이미지는 `html2canvas` 또는 더 가벼운 DOM-to-image 후보 비교 |

추천 테이블 초안:

```text
opponents
  id, name, area, uniform_color, public_note, created_at, updated_at

match_attendance
  id, match_id, player_id, status, desired_position, mercenary_count?, updated_at

match_arrivals
  id, match_id, player_id, arrived_at, desired_position, updated_at

match_comments
  id, match_id, author_user_id, body, created_at, deleted_at

club_settings
  key, value_json, updated_at
```

주의:

- `match_comments`, `club_settings` 중 회칙/운영 메모는 public read 대상에서
  제외하는 것이 안전하다.
- `player_id`가 없는 일반 방문자는 참석 투표를 할 수 없게 하는 편이 현재
  권한 모델과 맞다.

## 보안 관점

### Samala에서 주의할 점

Firebase 설정값이 브라우저에 있는 것 자체는 일반적으로 비밀 노출이 아니다.
Firebase/Supabase의 public key는 클라이언트 앱에서 사용할 수 있다. 실제
보안 경계는 Firebase Database Rules, Storage Rules, Auth 정책이다.

다만 코드 구조상 다음 점은 주의가 필요하다.

| 항목 | 관찰 | 위험 |
| --- | --- | --- |
| 클라이언트 직접 쓰기 | 브라우저 JS에서 Realtime Database 경로에 직접 `set`, `update`, `remove` | Rules가 약하면 UI를 우회한 쓰기 가능 |
| UI 권한 의존 | `window.isAdminOrStaff`, `window.userRole`로 버튼 표시/저장 분기 | 최종 보안이 Rules에 없으면 취약 |
| 개인정보 저장 | 출생년도, 연락처, 사진, 회비 비고 사용 | 공개 조회 또는 권한 오류 시 민감 정보 노출 |
| 공개 가입/익명 | register, Google login, anonymous sign-in 존재 | 운영 대상이 넓어져 승인/스팸/권한 관리 부담 증가 |
| 초대 토큰 | `inviteTokens/{token}: true` 형태 사용 확인 | 만료/회수/사용 횟수 정책이 없으면 링크 장기 유효 가능 |
| 동적 HTML | 많은 `innerHTML` 사용. 댓글은 일부 escape 처리 확인 | 모든 사용자 입력이 escape되지 않으면 XSS 위험 |
| 보안 헤더 | CSP 등은 응답 헤더에서 확인되지 않음 | 외부 CDN/inline script가 많아 CSP 적용 난도가 높음 |

### 우리 앱의 권장 보안 원칙

현재 우리 앱의 공개 조회 정책을 유지한다면:

1. 공개 페이지에는 선수 연락처, 출생년도, 사진, 회비, 내부 메모를 넣지 않는다.
2. 참석/희망 포지션은 공개해도 되는지 먼저 결정한다. 기본값은 public read가
   아니라 editor/member-only read가 더 안전하다.
3. 회비 기능을 만들 경우 public schema에 넣더라도 RLS `SELECT`를 공개하지 않는다.
4. 댓글/공지/회칙은 HTML로 렌더링하지 말고 plain text로 저장/표시한다.
5. 모든 쓰기는 기존처럼 Server Action 권한 체크와 Supabase RLS를 함께 사용한다.
6. 결과 입력 권한과 운영 권한은 계속 분리한다.

추천 기본값:

- `team_editors`만으로 모든 새 기능을 열지 말고, 기능별 권한을 나눈다.
- 예: `can_manage_match_results`, `can_manage_attendance`, `can_manage_private_members`.
- normal editor에게 라인업/참석 관리는 허용하되, 회비/연락처는 owner 전용으로
  시작하는 편이 안전하다.

## 디자인 관점

### 참고할 점

Samala는 모바일 사용자가 빠르게 누를 수 있는 구조가 많다.

- 상단/탭 기반 이동
- 다음 경기 중심 홈
- 큰 팝업으로 작업 집중
- 아이콘이 있는 짧은 버튼
- 경기 카드에서 참석, 미션, 전술, 공유로 바로 접근
- 모바일 설치 안내
- 이미지 공유용 전용 카드 디자인

우리 앱에 특히 참고할 부분:

- `/schedule` 첫 화면을 캘린더보다 “다가오는 경기 + 참석 현황” 중심으로 시작
- `/lineup`에서 match roster를 직접 고르는 것 외에 참석자 기반 추천 제공
- 경기 완료 후 결과 카드/라인업 카드 공유 버튼 추가
- 선수 개인 상세에 최근 기록, 포지션 분석, 성장 추이 같은 요약 제공

### 그대로 따라 하지 않을 점

Samala는 기능이 많아 한 화면 안에 카드와 팝업이 많이 쌓인다. 우리 앱은
운영 도구 성격이 강하므로 다음 원칙을 유지하는 편이 좋다.

- 핵심 작업 화면은 라인업, 경기, 선수, 랭킹처럼 명확히 분리한다.
- 카드 안에 카드가 겹치는 구조를 줄인다.
- 관리 기능은 메뉴/권한별로 분리해서 “보이는 것”과 “할 수 있는 것”을 명확히 한다.
- 부정적 랭킹은 기본 대시보드에 노출하지 않는다.
- 모바일에서 버튼/텍스트가 줄바꿈되어 화면을 밀지 않도록 고정 크기와 overflow를 설계한다.

## 제품 로드맵 제안

### 1단계: 경기 전 참석 관리

제품 효과:

- 라인업 준비 전에 실제 참석자와 희망 포지션을 볼 수 있다.
- match roster 구성 시간이 줄어든다.

구현 범위:

- `match_attendance` 테이블
- 참석 상태: `attending`, `absent`, `maybe`
- 희망 포지션 선택
- `/schedule` 또는 `/lineup`에서 경기별 참석 요약 표시
- editor는 전체 수정 가능, 본인 참여자는 자기 상태만 수정 가능 여부는 별도 결정

남은 리스크:

- 현재 앱에는 일반 선수 로그인 모델이 없다. 선수 본인 입력을 허용하려면
  member account 개념이 필요하다.
- 단기 대안은 editor가 참석 상태를 대신 입력하는 방식이다.

### 2단계: 상대팀 관리

제품 효과:

- 상대팀별 전적과 스타일을 축적할 수 있다.
- 경기 생성 시 상대팀 이름 입력 오류가 줄어든다.

구현 범위:

- `opponents` 테이블
- 경기 생성/수정에서 opponent 선택
- 기존 `matches.opponent` 문자열은 마이그레이션 계획 필요
- 상대팀별 승/무/패, 득점, 실점, 최근 경기 표시

### 3단계: 공유 이미지

제품 효과:

- 라인업과 결과를 카카오톡/인스타그램 등으로 쉽게 공유할 수 있다.
- 모바일 캡처보다 일관된 이미지가 나온다.

구현 범위:

- 라인업 캡처 카드
- 경기 결과 카드
- 다운로드 및 Web Share API

주의:

- 공개 이미지에 개인정보나 내부 메모가 포함되지 않게 한다.
- 이미지 생성용 DOM은 화면 UI와 분리해 고정 비율로 만든다.

### 4단계: private 운영 기능

제품 효과:

- 회비, 연락처, 회칙, 내부 공지를 앱 안에서 관리할 수 있다.

선행 조건:

- public read와 private read 분리
- RLS 검증
- owner/admin 전용 권한
- 개인정보 보관 목적과 삭제 정책 결정

## 최종 추천

Samala에서 가장 먼저 가져올 가치는 “팀 운영의 현장성”이다. 지금 우리 앱은
데이터 구조와 라인업 기록은 잘 잡혀 있지만, 경기 당일 전후의 운영 흐름은
아직 비어 있다. 따라서 다음 작업은 `참석 관리 + 희망 포지션 + 상대팀 관리`
순서가 가장 실용적이다.

보안과 유지보수 측면에서는 Samala의 구현 방식을 그대로 복제하지 않는 것이
맞다. 우리 앱은 이미 Supabase RLS, Server Actions, 권한 분리 문서가 있으므로
새 기능도 그 구조 안에 넣어야 한다. 특히 연락처, 출생년도, 사진, 회비, 댓글,
회칙은 공개 조회 정책을 깨기 쉬우므로 별도 private 기능으로만 검토한다.
